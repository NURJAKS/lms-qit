ЧАСТЬ 2. ПРОЕКТИРОВАНИЕ ВЕБ-ПРИЛОЖЕНИЯ «QAZAQ IT ACADEMY»

2.1 Разработка структуры и архитектуры приложения

Веб-приложение «Qazaq IT Academy» представляет собой образовательную платформу класса LMS (Learning Management System — система управления обучением) с интегрированным AI-помощником и элементами геймификации. Продукт ориентирован на организацию дистанционного и смешанного обучения IT-дисциплинам (в том числе программированию, веб-разработке, смежным направлениям) и реализован по клиент-серверной архитектуре: серверная часть предоставляет REST API, клиентская — интерактивный интерфейс в браузере.

Технологический стек актуальной версии проекта включает следующие компоненты.

Серверная часть (Backend): язык Python, фреймворк FastAPI; ORM SQLAlchemy; валидация и схемы данных — Pydantic; аутентификация на основе JWT; хеширование паролей (passlib/bcrypt); ограничение частоты запросов — SlowAPI; интеграция с внешними LLM — OpenAI API и Google Gemini (библиотека google-generativeai); дополнительно — генерация отчётов и документов (в том числе openpyxl, reportlab, Pillow), планировщик задач APScheduler.

Клиентская часть (Frontend): фреймворк Next.js 16 (App Router), библиотека React 19, язык TypeScript; стилизация — Tailwind CSS 4; управление серверным состоянием и кэшированием — TanStack Query (React Query); глобальное клиентское состояние — Zustand; формы — react-hook-form совместно с Zod; анимации — библиотека Motion; форматированный ввод контента у преподавателя — TipTap; визуализация данных — Recharts; HTTP-клиент — axios.

Хранение данных: в контейнеризированном развёртывании используется СУБД PostgreSQL (сервис в Docker Compose); для локальной разработки допускается конфигурация с SQLite через переменную окружения DATABASE_URL. Загружаемые файлы (в том числе медиа и сгенерированные сертификаты) сохраняются в каталоге uploads серверной части.

Интернационализация: интерфейс поддерживает три языка — русский, казахский и английский; строки интерфейса централизованы (например, в модуле переводов), что обеспечивает согласованность локализации.

Развёртывание: предусмотрен запуск через Docker Compose (frontend, backend, PostgreSQL) и локальный запуск backend/frontend отдельными сценариями из корня репозитория.

Ролевая модель включает шесть типов пользователей: администратор, директор, куратор, преподаватель, родитель и студент. Для каждой роли формируется свой набор доступных разделов и виджетов; критичные операции на сервере дополнительно защиются проверкой роли и прав доступа, что исключает обход ограничений только средствами клиента.

Ниже перечислены ключевые функциональные модули и экраны системы.

1) Главная страница (Landing Page) — точка входа для неавторизованных пользователей: маркетинговый блок (hero), статистика платформы, преимущества, демонстрация технологий и курсов, каталог курсов с фильтрацией по категориям, отзывы, FAQ, контакты; в шапке — переключение языка и темы, переход в личный кабинет.

2) Авторизация — вход по электронной почте и паролю с валидацией на клиенте, отображение ошибок, опционально «запомнить меня»; JWT сохраняется и используется в заголовке Authorization последующих запросов.

3) Регистрация — в архитектуре системы предусмотрено создание учётной записи с проверкой данных; в конкретной конфигурации развёртывания регистрация может быть ограничена политикой организации (например, создание пользователей администратором).

4) Дашборд студента — обзорные карточки статистики, приветствие, блок «продолжить обучение», геймификация (ежедневные задания, мотивация через рейтинг), AI-челлендж, дедлайны, сообщество, быстрые ссылки на курсы.

5) Дашборд административных ролей (администратор, директор, куратор) — аналитика, управление пользователями, курсами, тестами, отзывами, магазином и заказами, курьерами, обзорные метрики.

6) Кабинет преподавателя — группы и студенты, задания с критериями оценивания (рубрики), проверка работ, загрузка материалов, rich text при оформлении заданий.

7) Кабинет родителя — прогресс привязанных детей, успеваемость, сравнение с группой (в рамках прав доступа API).

8) Страница курса и темы — описание курса, структура тем, визуализация пути обучения, теория и видео, заметки к темам, видеоплеер с ограничениями для режима обучения (в том числе антисписывание при просмотре), итоговое тестирование.

9) AI-помощник — плавающий чат-виджет для авторизованных пользователей: ответы по материалу, подсказки, учёт лимитов и контекста (в том числе режим прохождения теста), журналирование запросов, анти-чит.

10) Магазин (Q-Market) — каталог товаров за внутриигровую валюту (Q-coins), корзина, адрес доставки, подтверждение покупки, администрирование заказов.

11) Таблица лидеров — ранжирование студентов, выделение лидеров, фильтры и наглядное представление позиций.

12) Расписание и календарь — календарный виджет, события, детали мероприятий.

13) Сообщество — лента, взаимодействия, групповой чат (в рамках реализованного функционала проекта).

14) Профиль — данные пользователя, аватар, язык и тема, выход из системы.

Модульная архитектура с чётким разделением Frontend/Backend и контрактом REST API снижает связность компонентов, упрощает сопровождение и позволяет независимо развивать отдельные подсистемы (обучение, магазин, AI, аналитика).

2.2 Процесс разработки макета веб-приложения

Проектирование пользовательского интерфейса (UI) и пользовательского опыта (UX) выполнялось с ориентацией на современные образовательные платформы: читаемая типографика, контрастные акценты, поддержка светлой и тёмной темы, адаптивная вёрстка под мобильные, планшетные и десктопные разрешения.

Цветовая палитра опирается на сочетание тёмно-синих и насыщенных акцентных оттенков (голубой, розовый) в градиентах кнопок и баннеров; для светлой темы используются нейтральные фоны и мягкие тени, для тёмной — приглушённые серые поверхности. Для категорий курсов задаются различимые акцентные цвета, что облегчает навигацию в каталоге.

Типографика и сетка выстроены с опорой на утилиты Tailwind CSS; контрольные точки (breakpoints) sm/md/lg/xl используются для адаптации сеток, размеров шрифтов и отступов. Интерактивные элементы, предназначенные для касания, проектируются с достаточной областью нажатия.

Главная страница задаёт визуальный тон продукта: динамичный hero-блок, анимированная типографика, блоки доверия (статистика, отзывы), интерактивные секции (карусель, бегущая строка, подбор курса). Внутреннее приложение после входа опирается на боковую навигацию для широких экранов и нижнюю панель — для узких, плюс верхнюю панель с поиском и уведомлениями.

Персонализация включает переключение языка интерфейса и темы; состояние авторизации и профиля синхронизируется с глобальным хранилищем и при необходимости с локальным сохранением для устойчивости сессии при перезагрузке страницы.

2.3 Реализация системы аутентификации

На сервере реализованы эндпоинты модуля `/auth` (вход и, при включённой политике, регистрация). Пароли не хранятся в открытом виде: применяется хеширование. Выдача и проверка доступа строятся на JWT: токен передаётся клиентом в заголовке Authorization, на сервере извлекается идентификатор пользователя и роль.

Для снижения риска brute-force на эндпоинты аутентификации накладывается rate limiting (например, ограничение числа запросов с одного IP в единицу времени) средствами SlowAPI. Успешные входы и значимые действия могут протоколироваться в журнал активности для аудита.

На клиенте формы строятся на react-hook-form с схемами Zod: проверяются формат email, длина пароля и другие правила до отправки на сервер. Ошибки API преобразуются в понятные пользователю сообщения с учётом выбранного языка. После успешного входа выполняется запрос к эндпоинту профиля текущего пользователя (`/users/me` или аналог в проекте) для получения полных данных (роль, имя, аватар и т.д.) и заполнения состояния приложения.

2.4 Реализация главной страницы (Landing Page)

Главная страница реализована в приложении Next.js с гибридным подходом к рендерингу: интерактивные секции оформляются клиентскими компонентами с использованием хуков React для анимаций, наблюдателей видимости и локального состояния.

Загрузка каталога курсов выполняется запросом к API; для кэширования и повторного использования данных применяется TanStack Query. Фильтрация по категориям может выполняться на клиенте по уже загруженному списку, что даёт мгновенный отклик интерфейса без лишних обращений к серверу.

Hero-секция сочетает фоновую графику, оверлей для контраста, анимированный текст и призыв к действию (переход к курсам или в кабинет). Блоки статистики могут использовать анимацию счётчика при появлении в зоне видимости. Каталог отображается адаптивной сеткой карточек: обложка, категория, краткое описание, метаданные, цена, кнопка перехода.

2.5 Реализация дашборда студента

Дашборд агрегирует данные из нескольких независимых API-запросов (записи на курсы, детальный прогресс, список курсов и др.), которые по возможности загружаются параллельно, чтобы сократить время до первого отображения.

Виджеты включают: карточки KPI (завершённые курсы, баллы, суммарный прогресс, число записей); баннер приветствия; «Продолжить просмотр» с привязкой к сохранённому прогрессу видео/тем; ежедневные задания с прогресс-барами и кнопкой получения награды; мотивацию через позицию в лидерборде; карточку AI-челленджа; дедлайны и блок сообщества. Появление блоков может сопровождаться анимацией (Motion) для улучшения восприятия загрузки страницы.

Маршрутизация внутри приложения после входа различает роли: студенту показывается описанный набор виджетов, преподавателю и административным ролям — соответствующие панели.

2.6 Реализация экранов курсов и учебных материалов

2.7 Реализация системы тестирования

Серверный модуль тестов обеспечивает выдачу списка тестов и вопросов, приём ответов, подсчёт результата и сохранение итогов в модели прогресса. Клиентский компонент теста отображает вопросы с вариантами ответов, навигацию «Назад/Далее», индикатор прохождения; по завершении формируется экран результатов с разбором (корректные и выбранные ответы).

Успешное прохождение итогового теста с набором выше порога используется как условие для выдачи сертификата о прохождении курса (логика сертификата реализована на backend с генерацией изображения/PDF в зависимости от конфигурации).

2.10 Реализация AI-помощника и анти-читерства

Плавающий виджет чата отправляет сообщения на защищённый эндпоинт AI-модуля. На сервере выполняется ряд проверок: лимиты использования (в том числе дифференциация для премиум-пользователей при включённой бизнес-логике), определение контекста (например, активный тест — тогда модель получает инструкции не выдавать прямые ответы на задания), эвристический анализ «подозрительных» формулировок (паттерны запросов готовых ответов на тест).

История обращений может сохраняться в базе с признаками подозрительности для последующего анализа администратором. Генерация ответа выполняется через выбранного провайдера LLM (OpenAI и/или Gemini) с системным промптом, учитывающим курс и режим безопасности.

2.11 Реализация магазина Q-Market и геймификации

Виртуальная валюта (Q-coins) начисляется за обучающую активность, ежедневные задания, достижения и участие в рейтингах; списание происходит при покупках в магазине. 2.13 Управление состоянием клиентского приложения

Глобальное состояние аутентификации реализовано через Zustand с персистенцией выбранных полей (токен, пользователь) в localStorage с учётом особенностей Next.js (проверка доступности window). API-клиент подставляет токен в заголовки запросов.

Серверные данные (курсы, дашборд, магазин) кэшируются TanStack Query с ключами запросов и инвалидацией после мутаций (например, после покупки или завершения урока), что обеспечивает актуальность UI без полной перезагрузки страницы.

3.1 Установка и запуск приложения

Ниже приведены требования к окружению и шаги запуска для разработчика или администратора, разворачивающего систему локально или в Docker.

Системные требования:
— Python 3.12 или новее (рекомендуется);
— Node.js 20.9.0 или новее (согласно engines в frontend-next/package.json);
— Git;
— Docker и Docker Compose — для контейнерного запуска с PostgreSQL.

Шаг 1. Получение исходного кода.
Клонировать репозиторий проекта и перейти в корневую директорию (команды зависят от фактического URL репозитория организации).

Шаг 2. Запуск через Docker Compose (рекомендуемый способ для полного стека).
В корне проекта выполнить:
    docker compose up -d --build
После сборки контейнеров:
— backend доступен на http://127.0.0.1:8000 (или http://localhost:8000);
— frontend — на http://127.0.0.1:3000;
— PostgreSQL проброшен на порт 5432 (параметры пользователя/пароля/БД задаются в docker-compose.yml).

Документация OpenAPI (Swagger) обычно доступна по адресу http://127.0.0.1:8000/docs.

Шаг 3. Локальный запуск backend без Docker.
Перейти в каталог backend, создать виртуальное окружение, установить зависимости:
    cd backend
    python3 -m venv .venv
    source .venv/bin/activate   (в Windows: .venv\Scripts\activate)
    pip install -r requirements.txt
Скопировать файл конфигурации окружения из примера (если имеется .env.example) в .env и задать DATABASE_URL, SECRET_KEY, ключи API для LLM при использовании AI.

Инициализация и заполнение демонстрационными данными (если в репозитории предусмотрены сценарии):
    python init_db.py
    python seed_data.py
    python seed_shop.py
(наличие и имена скриптов уточнять по актуальной документации репозитория.)

Запуск сервера разработки:
    python -m uvicorn app.main:app --host 127.0.0.1 --port 8000 --reload

Шаг 4. Локальный запуск frontend.
В отдельном терминале:
    cd frontend-next
    npm install
    npm run dev
Приложение откроется на http://localhost:3000. Переменные окружения Next.js (базовый URL API и прочие NEXT_PUBLIC_*) задаются в .env.local согласно README проекта.

Шаг 5. Скрипты быстрого запуска из корня (если присутствуют).
    ./run-backend.sh
    ./run-frontend.sh
либо единый сценарий start.sh при его наличии.
